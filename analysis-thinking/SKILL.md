# Analysis Thinking — 分析分诊器

**母句（常驻总纲）**：分析不是套框架，而是在不完备信息中，先识别对象类型，再根据对象与目的选择切入点，控制切入点数量，识别关键断点与依赖关系，通过解耦或重组建立可判断、可验证、可沟通、可行动的临时解释结构。

切入点决定从哪里看；断点决定在哪里切；依赖决定为什么连；解耦决定怎么改。

## 分诊路由表（轻量索引：每次只加载一份，禁止全量加载）

| 信号 | 加载 | 提供什么 |
|---|---|---|
| 对象轻 / 时间紧 / 首次探索 | [simple.md](simple.md) | 八型分型 → 1-3个切入点 → 断点 → 最小验证动作 |
| 复杂对象 / 要闭环到行动 | [standard.md](standard.md) | 在简版之上加：依赖耦合、解耦重组、行动承接 |
| 已确认系统型/组织/业务/产品 | [system.md](system.md) | 模块-接口-依赖-耦合-解耦专用刀法 |
| 分析已完成，要整理汇报表达 | [mckinsey-polish.md](mckinsey-polish.md) | 表达整理器：结构化措辞（不用于找断点） |

判型拿不准 → 默认 simple.md 先跑一轮，再决定是否升级 standard。

## 流程

1. 读材料，判断对象八型（可多选须排序）与场景紧急度；材料不足以分型时先返问（≤2 个问题），不硬分。
2. 按路由表 **Read 唯一一份**子文件，按其指令执行分析。
3. 输出必含五要素：对象分型（含分错的代价）/ 切入点（选了什么、暂弃什么）/ 关键断点 / 简要判断 / 下一步最小验证动作。

## 硬性断言（自检）

- 单次执行加载子文件 ≤1；确需升级必须明说"从简版升级到标准版，因为…"
- 输出五要素齐全，且事实/推断/假设/价值判断四分标注至少出现一次
- 每个断点都有一句"为什么它是断点"——不许光列名词，排版整齐不等于分析

## 边界

- 数据集/指标类交给 da-* 系列（规划中）；架构可视化交给 c4-diagram；纯概念解剖交给 ljg-learn。
- 本 skill 输出的是"临时解释结构"，不是最终结论——结论要等最小验证动作跑完。
