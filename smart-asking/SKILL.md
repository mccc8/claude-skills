# Smart Asking — 求助分诊器

**核心信念**：提问不是找答案，是一次知识外化——好问题的第一受益人是提问者自己；解决后不沉淀，不算闭环。

## 第零步（永远先做）：三分钟自查

报错原文检索 + 手册对应章节 + **动手做最小复现**。做复现的过程经常让问题自解（小黄鸭效应）；没做过自查的求助，任何路由都不接。

## 路由判定：三轴（每次判定必须三轴各给一句理由）

**可验证性**（谁能判答案对错）× **上下文归属**（关键信息在公共域还是组织/个人域）× **代价结构**（错误答案的代价 vs 等待的代价）

| 信号 | 路由 | 加载 |
|---|---|---|
| 答案可自验（跑一下便知）/ 上下文能完整贴出 / 公共稳定知识 / 要方案草稿 | **问 AI**（默认第一站，便宜） | [ask-ai.md](ask-ai.md) |
| 错误代价不可逆或验证昂贵 / 关键信息在组织隐性域（口径、历史坑、"当年为什么这么设计"）/ 需要有人对答案负责 / AI 两轮仍打转 | **问人** | [ask-human.md](ask-human.md) |
| 疑似上游 bug（维护者才能修）/ 答案值得公共沉淀（别人也会撞） | **发公开**（issue/社区） | [ask-public.md](ask-public.md) |
| 问题已解决 | **沉淀**（强制） | [settle.md](settle.md) |

拿不准 → 默认先 AI（算力廉价）；但**"不可逆"信号一旦出现，强制问人**，AI 只许当参谋不许当决策者。

## 流程

1. 确认自查已做（没做就先陪用户做，不代跳）。
2. 三轴判定路由，**先输出"问谁 + 三轴理由"**，再加载对应子文件（每次 ≤1 份）生成提问稿。
3. 问题解决后加载 settle.md 完成沉淀——这一步不做，主动提醒欠账。

## 硬性断言（自检）

- 路由判定先于提问稿，且三轴理由各 ≥1 句
- 提问稿必含：[对象]-[偏差] 式标题 / 目标先于步骤 / 最小复现或最小上下文
- 问人的稿必含研究日志 ≥2 条（展示而非声称）；问 AI 的稿必含"不确定请标注"+ 验证计划
- 会话内出现"已解决"信号而未做沉淀 → 必须提示

## 边界

- 一事一议；agent 编排/loop 设计不在此列。
- 分析问题结构 → analysis-thinking；先想清目标 → goal-builder。
