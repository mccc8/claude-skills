# 双向同步机制（Dual Sync）

> 一句话总纲：**GitHub 是 registry，不是运行的脐带。** 云端管发布与版本，本地管执行与低摩擦。

## 角色分工

| 角色 | 承担 | 位置 |
|---|---|---|
| **发布源**（事实来源） | 版本、历史、回滚、分享、协作 | github.com/mccc8/claude-skills（公开）＋ workdeck（私有，四资产数据） |
| **执行源** | Claude Code / Codex 运行时实际读取 | 本机 `~/.claude/skills/` 与 `~/WorkDeck/` |
| **管理入口** | 浏览、编辑、一键上/下行 | WorkDeck（127.0.0.1:8766）/ SkillDeck（:8765） |

## 上行（本地 → GitHub）

任一入口等价，幂等可重复：

1. WorkDeck / SkillDeck 顶栏「⬆ 发布」按钮
2. 对 Claude 说"发布/同步到 github"
3. 命令：`bash ~/.claude/skills/github-publish/publish.sh ~/.claude/skills claude-skills`

上行自动完成：暂存→提交（带时间戳）→推送；技能仓库还会**自动重建展示页**（scripts/build_page.py）。

## 下行（GitHub → 本地）

1. WorkDeck / SkillDeck 顶栏「⬇ 拉取」按钮
2. 对 Claude 说"从 github 拉取"
3. 命令：`bash ~/.claude/skills/github-sync/sync-down.sh`（全仓库拉到 `~/GitHub/` + 索引页）

## 冲突裁决（政策，2026-07-06 定）

**完成双向同步后，本地与线上有纠纷，一律以 GitHub 线上版本为准。**

执行细则（防误伤）：

- 工具**默认不自动丢弃**本地改动：下行遇到本地未发布修改会拒绝并提示先发布。
- 触发"以线上为准"的覆盖，需要用户说一句确认；执行时先把本地版备份为 `.bak-日期` 再覆盖。
- 永不 force push；换机器/新环境，第一动作永远是拉取。

## 日常节奏（最省心版）

- 改完 skill → 点一次「发布」（或攒到当天收工一次）。
- 每周一 10:01 `weekly-skill-audit` loop 自动体检全部 skill 触发质量并出报告。
- 新增 skill 后展示页无需手动管——发布时自动重建。

## 演进预留（与三阶段定位对齐）

- Registry（当前）：本文件所述全部。
- Personal Hub（下一步）：每个 skill 增加 manifest（版本/状态：实验-稳定-废弃/最近使用），WorkDeck 读取展示。
- Pack Manager（远期）：`npx skills add mccc8/claude-skills` 已可用作他人安装入口；后续补锁版本与依赖声明。
