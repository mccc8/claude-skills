# 仓库起源与定位

> 蒸馏自 2026-07 前后关于 WorkBuddy 政务版与个人 Skill 体系的两场长对话。原始记录存于私有空间，此处只留决定仓库走向的结论。

## 起源：从 WorkBuddy 看到的东西

广东"湾擎·WorkBuddy"政务版引出的判断链：

- 个人版解决**个体效率**，企业版解决**组织协作**，政务版解决**组织治理**——三种不同的问题。
- 企业版真正卖四种组织能力：**方法复用、知识复用、产物复用、权限治理**。
- 分水岭一句话：**AI 产出的价值，是否能离开个人（会话），进入组织（资产）循环。**

把"组织"换成"个人"，就是本仓库的存在理由：让 AI 产出离开会话，进入个人资产循环。四类资产 = 计划 / 知识 / 产物 / 技能，本仓库承载其中的**技能**层。

## "多此一举吗？"——定位裁决

曾自我质疑：本机 skill 上传 GitHub、再做本地软件同步回来，是否绕圈。裁决结论：

- 如果只是"个人同步"，确实略重（给水杯造物流系统）。
- 但定位是 **Skill Registry（技能注册表）**：GitHub 不是网盘，而是 skill 的**版本化资产库**——版本、历史、回滚、分享、协作、审计都白送。
- 演进路径锁定为三阶段，**不许跳级**：

| 阶段 | 名字 | 边界 |
|---|---|---|
| 当前 | Registry 登记系统 | 能找到、能更新、能回滚，＋知道它适合干什么 |
| 下一步 | Personal Hub | manifest 化、状态分层（实验/稳定/废弃）、使用反馈 |
| 远期 | Pack Manager | 一键安装/锁版本/依赖声明，向他人分发 |

## 架构分工（一句话记住）

**GitHub 可以是 Skill 的 registry，不应该是每次运行的脐带。**

- GitHub = 发布源 / 版本源 / 协作源（事实来源，纠纷以线上为准）
- 本机 `~/.claude/skills/` = 执行源 / 缓存源（运行永远读本地）
- 本地软件（WorkDeck / SkillDeck）= 管理入口 / 同步入口 / 浏览入口

## 防跑偏负清单（对自己的约束）

- 不做过早治理化：权限系统、审批流、评分市场——"把自己从使用者变成运维奴隶"。
- 不做大 Skill 社区，不证明 Skill 概念伟大。
- 同时在制维护的 skill ≤ 3 个。
- 收尾金句：**Skill 的价值不在把一句提示词保存下来，而在把一套反复有效的方法，变成可验证、可维护、可复用的工作资产。**
