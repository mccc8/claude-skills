# 从口号到方法论：一个人的 Skill 写作探索史

> 基于 2026-03-01 至 2026-05-25 期间 21 条 `#project/skill复利计划` flomo 笔记的复盘与萃取。

## 〇、一句口号的诞生与死亡

2026 年 3 月 1 日晚上 10 点半，这个计划以一句口号开场：

> 每天看一个 skill，每天写一个 skill，每天用一次 skill。

这句口号活了大约一周。3 月头七天贡献了全部 21 条笔记中的 17 条，之后整个 3 月中旬只剩一条，然后是漫长的沉默，直到 5 月 25 日最后一条笔记像回光返照一样出现——而恰恰是那条笔记，给出了全计划最可复用的结构。

这不是失败史。日更断了，但认知留下来了。复盘这 86 天，真正的产出不是"写了多少个 skill"，而是三次认知跃迁和一套可以继续用的方法论。这篇长文就是把它们从流水账里挖出来。

## 一、第一跃迁：Skill 不是知识文档，是 Trigger + Action

Day 1 看的第一个 skill 是 vercel-labs 的 find-skills。当时的笔记里有一句话，现在看是整个计划里最锋利的原创结论：

> **Skill = Trigger + Action，不是知识文档。写 Skill 要从用户行为逆向设计触发条件，不是从功能出发正向描述。**

这句话为什么重要？因为几乎所有人（包括当时的我）写 skill 的本能是写"这个 skill 能干什么"——一份功能说明书。但 find-skills 写得好的地方完全不在功能，而在它的触发设计：

- **兜底逻辑**：找不到 → 直接帮做 → 提示可自建。元功能永不失败。
- **示例驱动**：六种句式覆盖直接、间接、模糊、情绪化四类表达。尤其捕捉了"用户根本不知道 skill 这个词"和"只表达困境不表达需求"的隐性场景。

由此触发了一个更深的观察：**人类并不主动搜索技能，是场景线索触发内隐记忆。** 你遇到问题时不会去"查工具列表"，而是大脑里先闪现某个线索。所以外脑（skill 库）的设计目标不是"可搜索"，而是"像内脑一样自然浮现"——这就要求 description 写的不是功能清单，而是场景线索。

一个具体的应用案例：想给"不可能三角"写个 skill，正确写法不是解释什么是不可能三角，而是：

- Trigger：用户描述"既要、又要、还要"的困境
- Action：列出三个目标 → 强制划掉一个 → 针对剩余两个给建议

## 二、第二跃迁：Skill 存在的前提，是否定 Human-in-the-Loop

3 月 3 日，读 Anthropic 的万字 Skills 教程时，一段视频片段和教程内容对上了：

> "人在回路"审查模式已成为系统瓶颈。Human in the loop is the constraint。每个 PR 都等人看？等不起。纠正成本低，等待成本高。

于是他们移除 human-in-the-loop，代之以**渐进式披露的文件系统**：一个 100 行的 AGENTS.md 当目录，指向结构化的设计文档、架构规范、执行计划。让模型自己狂飙，按需查阅，跑崩了读日志，改了再跑。

当天笔记里的结论，是整个计划认知深度的顶点：

> **Skill 的核心就是：否定 HITL 的设计前提才会需要 skill。如果要 skill，就要做到渐进式披露、按需加载。至于其他内容，都是技巧问题。**

换句话说：如果你打算每一步都盯着 AI，你不需要 skill，你需要的是聊天。skill 是写给"你不在场的时候"的——它必须预判场景（Trigger）、自带判断标准（Action 里的约束）、并且短到不污染上下文（渐进式披露：SKILL.md 是目录，细节放引用文件里按需加载）。

这也解释了第一跃迁：为什么 Trigger 比功能描述重要？因为你不在场，没人帮 AI 决定"现在该用这个了"。

## 三、第三跃迁：没有评估的 skill 只是许愿

同一天看官方 skill-creator 的笔记里，记下了三个概念：

- **Baseline 对比**：新 skill 要和"无 skill 的裸模型"对比；改进版要和旧版对比。没有基线，"变好了"只是感觉。
- **断言（Assertion）**：用客观可验证的规则自动判断输出是否合格——包含特定要素？格式正确？长度达标？好的断言避免主观判断。
- **Grader**：评分子代理，把"满不满意"变成可重复的打分。

> 技能的评估和测试，可以看作提示工程的一个更高级、更系统化、更自动化的版本。

这一条在 3 月 2 日有过一次诚实的阴性实验佐证：同样的 prompt 和上下文，用 skill 和不用 skill，在 manus、coze、kimi 上跑，**差别并不大**。当时的归因是"可能 skill 写得不好"——现在回头看，真正的教训是：没做 baseline 设计和断言的实验，连"skill 有没有用"都无法回答。那天笔记里还有半句先见之明："真正的瓶颈是自动路由能不能成立"——自动路由，正是 Trigger 设计的工程化说法。

## 四、中间产物：三个值得留下的实践

认知之外，86 天里有三个实践成果通过了"效果成果"检验：

**1. 白板讲解图 prompt（3 月 4 日）。** 一段很简约的提示词——手绘风格流程图、黑红蓝三色马克笔、涂改和箭头、16:9、放不下就拆最多三张——就把斯坦福 AI 创业潮研究和《Some Simple Economics of AGI》两篇文章变成了五张白板讲解图。它证明了：**好 skill 往往极短，靠的是精确的风格约束而非冗长说明。**

**2. "休会"协议（3 月 17 日）。** 把 AI 对话的中断从"信息丢失"变成"强制收敛"：将当前对话压缩为最小可续状态——核心问题一句话、思维推进 3-5 个节点（A→B 格式）、已排除路径最多两条、悬而未决 2-3 条、结束判定三选一（收敛/发散/停滞），总长 ≤150 字，禁止复述、禁止补全、禁止空话。这条笔记的洞察在于重新定义了暂停："休会"本质不是为了重启，而是为了**强制收敛当前认知状态**，防止上下文污染。

**3. Goal 六要素（5 月 25 日，最后一条笔记）。** 让目标类 prompt 真正生效的前提结构：

| 要素 | 作用 | 盖楼比喻 |
|---|---|---|
| Objective | 一句话目标，禁"全部/彻底/improve"等虚词 | 盖一栋三层小楼，不是"把地搞好看点" |
| Scope | 作用边界 | 施工红线，邻居院子不许碰 |
| Constraints | 机械可识别的硬约束 | 不许劣质水泥、不许动承重墙 |
| Done when | 引用具体文件/命令的验收清单 | 水电通、消防过检、钥匙到手 |
| Stop if | 停止条件 | 挖到不明管线就停、超预算 20% 就停 |
| Token Budget | 资源上限 | 最多花 10 万，超了先停下汇报 |

注意 Stop if 和 Token Budget——大多数人的 prompt 只写"要什么"，从不写"什么时候停"。而这恰恰呼应了第二跃迁：AI 自主干活的前提，是你提前把刹车装好。

## 五、失败模式解剖：为什么日更死了

对流水账部分做个诚实的解剖，失败模式有四种：

**贪婪泛化。** 产品经理 skill 的笔记里自己就写了："要么泛化成设计，要么泛化成产品，甚至泛化到经理……得浅尝，得不贪，得收敛。" 诊断正确，但没治。阅读 skill 要追溯书摘、思考卡片、信息摄入场景——场景没简化，skill 就写不出来。**skill 的敌人不是难，是贪。**

**收藏代替消化。** 好几条笔记是纯链接堆积：pm-skills 两个仓库、Tiago Forte 读书法、systematic-debugging……存了，标了 tbd，然后没有然后。收藏产生的是"研究了"的幻觉。

**过程日志无残值。** skill creator 的七问七答沟通日志写了很长，但所有结论已经被最终 instruction 吸收。过程值得经历，不值得存档。

**平台漂移。** 3 月在 ChatGPT Project、coze、manus、notion 之间来回配置，每换一个平台，前面的沉淀就作废一部分。直到确定"得用 Claude"之后，这个损耗才停止。

而日更之所以断，根本原因是口号只有 Trigger（每天）没有 Action 的验收标准（什么算"写了一个"？）——**用第三跃迁的话说：这个计划本身没有断言。**

## 六、萃取：一套可以继续用的 skill 写作方法论

把三次跃迁和失败教训合并，得到五条操作性结论：

1. **先写 Trigger，后写 Action，不写知识。** description 里放的是场景线索和用户原话句式（包括模糊的、情绪化的、不知道术语的说法），不是功能列表。
2. **为"你不在场"而写。** 每个 skill 必须自带：边界（Scope）、硬约束（Constraints）、验收（Done when）、刹车（Stop if）。Goal 六要素不只是目标模板，它是所有 skill 的骨架。
3. **渐进式披露。** SKILL.md 保持一屏以内当目录，细节、示例、参考材料放引用文件，按需加载。长 skill 是上下文污染源。
4. **没有断言不算写完。** 每个 skill 至少配一个测试输入 + 三条客观断言 + 与裸模型的 baseline 对比。跑不过断言的 skill 回炉。
5. **收敛比完整重要。** 一个场景切干净，好过三个场景全覆盖。写不出来的时候，先删场景，再动笔。

## 七、尾声：复利到底复在哪

86 天，21 条笔记，最后留下 7 条核心资产和这一篇方法论。以日更 KPI 论，这个计划失败了；以复利论，它恰好成功在放弃日更的地方——因为复利的本金从来不是"skill 的数量"，而是**对 skill 这件事本身的认知压缩率**。

3 月 1 日的我以为要积累一个技能库；5 月 25 日的笔记证明，真正积累下来的是一套结构直觉：看到任何任务，条件反射地问——触发线索是什么？边界在哪？怎么验收？什么时候停？

这四问，就是 86 天买到的复利。
