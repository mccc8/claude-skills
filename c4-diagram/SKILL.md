# C4 Diagram — 架构信息图绘制

C4 的价值是**分层缩放**：不同受众看不同层。画图前先选层，不贪层。

## 流程

1. **选层**（最关键一步，只选 1-2 层）：
   - 给老板/新人/外部讲"这是个啥" → **L1 Context**（系统 + 外部用户/系统）
   - 讲部署单元、技术选型、数据流 → **L2 Container**（应用、服务、数据库、队列）
   - 讲某个容器内部模块职责 → **L3 Component**
   - L4 Code 层默认不画（代码即文档），用户明确要求才画。
2. **提取元素**：从用户描述中列出 人(Person)/系统(System)/容器(Container)/关系(Relationship)。信息不足的关系标虚线并注明"待确认"，不编造。
3. **绘制 SVG**：写一个自包含 `.svg` 文件到用户指定位置（默认当前目录 `c4-<系统名>-L<层>.svg`）。视觉规范见 [notation.md](notation.md)。
4. **自检断言**后交付，附一段 ≤5 行的图例说明。

## 硬性断言（自检，不过重画）

- 每个元素框内含三行：名称 / 类型标签（[Person]、[Container: 技术栈]）/ 一句职责描述
- 每条关系箭头都有动词标签（"调用/读写/发布"），无裸箭头
- 单图元素 ≤ 12 个；超了说明层选错了，拆层或合并
- 有图例（legend）区分颜色含义
- SVG 可独立打开（无外部依赖，字体用系统字体栈）

## 边界

- 一次只画当前选定的层；用户要"全套"时按 L1→L2 顺序分文件产出。
- 不做架构评审和建议（除非用户问）；图是描述，不是处方。
- 通用概念梳理/多中心发散内容 → 用 whiteboard-explain，不用本 skill（C4 只画软件系统结构）。
