# PRINCIPLES — 技能库宪法

> 总纲（交汇点）：**把正确的认知原则，通过稳定的方法，在现实中反复执行，并形成长期积累。**
> 三层定位：原则 = **质量约束**（防自欺——别把脑内噪声当事实；框架本身不保证诚实：DECIDE 列得出方案但保不了标准没偏，Pre-mortem 想得出失败但保不了没有选择性恐惧）；框架 = **场景化处理方法**（框架走流程，原则查污染）；工具 = **记忆与上下文脚手架**（外部化、连接、反馈）。
> 实体在两两相交处：原则×框架 = **受约束的决策流程**；原则×工具 = **可反思的思考记录**；框架×工具 = **可复用的决策上下文**。三圈中心不叫"判断力系统"——那是看见三角形就想修金字塔——叫 **deliberate thinking practice：可刻意练习的思考过程**，与总纲第一句同义（Good thinking is the result of deliberate practice）。
> **原则不是存下来供起来的，是要用起来的**——本文件的合法性每周由 loop 的原则回访检验：连续两周回访无话可说，即视为在供着，须修或删。

## 二、原则清单（每条都有出生证明——来自真实事件，不是抄来的）

### 写作原则（管一个 skill 怎么诞生）
1. **Skill = Trigger + Action，不是知识文档**——从用户原话逆向设计触发。
2. **为"不在场"而写**——自带边界、硬约束、验收、刹车（Stop if）。
3. **渐进式披露**——主文件 ≤60 行当目录，细节按需加载，每次只载一份。
4. **没有断言不算写完**——可脚本验证的必须脚本验证，禁止目测（出生证明：白板图目测"通过"后被脚本抓出三处失败）。
5. **收敛比完整重要**——在制 ≤3，先让三个活下来；25 个占位符不如 4 个真内容，除非像 untools 那样一次做完。

### 运行原则（管每次执行）
6. **先排除，后诊断**——物理预检 / 扭曲激励审计 / 提问前自查，三个 skill 同构：别用高深模型解释单纯的贫血。
7. **一次一问，一次一刀**——苏格拉底引导每次只问一个问题；分析每次只进一个工具。
8. **每层配最小行动**——不许停在"原来根源在价值观"的感悟里；改不动就回上层找次优解。
9. **事实四分**——事实/推断/假设/价值判断分开标注；坏数据必须认怂。
10. **说穿仍有效**——给他人设计的机制经得起当面说明；说穿就失效的是操纵。

### 演进原则（管系统怎么变老变强）
11. **新覆盖旧**——沉淀新结论必须标注废弃了哪条旧结论，只进不出的知识库是数字废墟（出生证明：untools 复原件被真原件取代删除）。
12. **重复即信号**——同一问题第三次出现，笔记升级为文档，文档升级为 skill，skill 升级为 loop。
13. **改尺子必须留痕**——断言校准要公示理由，且产物侧同步整改，不许只改尺子过关。
14. **线上为准**——双向同步后，纠纷以 GitHub 版本裁决（执行仍需人在场确认）。
15. **使用产生数据，数据修订原则**——本文件不是终稿；每条新原则须引用触发它的真实事件（本条自指）。

## 三、约束机制（谁管谁）

| 环节 | 约束物 | 执行者 |
|---|---|---|
| 立项 | 七条价值判定 ≥5 才动笔；在制 ≤3 | 人 |
| 写作 | 原则 1-5 + SKILL.md 结构断言 | Claude + 脚本 |
| 触发质量 | trigger-audit 五维评分 | weekly loop（每周一自动） |
| 评估 | EVALS.md 目标函数 + 诚实条款（漏测必须写明） | 脚本 + loop |
| 熵减 | settle 减法条款、原则 11 | 每次沉淀时 |
| 冲突 | 原则 14 | git + 人确认 |
| 宪法自身 | 原则 15：修订须附真实事件 | 人 + 月度回看 |

## 四、反馈回路与断环

```
Principles ──决定──▶ Frameworks ──编排──▶ Workflows ──执行──▶ Data/Documents
     ▲                                                            │
     └────────── 复盘修订 ◀── Agent 自动体检（weekly loop）◀───────┘
```

已通的环：使用 → 产物归档（WorkDeck）→ 周检报告 → 触发词/断言修订。
**断的环（诚实标注）**：产物 → 原则修订之间没有机制——weekly loop 只审触发质量，不问"本周哪条原则被验证/被违反"。已修：周检任务今日起新增该问项（见 changelog）。
仍断的环：复用率无统计（哪个 skill 被真实用了几次，无人记账）——列为 Personal Hub 阶段的 manifest 使命，不提前造轮子。

## 五、三层如何改写成"个人 KM → 组织 KM → Agent 工作流"

关键不是层的内容，是**每层的宿主搬家**：

- **原则住在人身上** = 个人知识管理。它死于遗忘，救法是本文件——把内化的直觉写成可引用的条文。
- **框架住在文档里** = 组织知识管理的最小单元。skill 公开仓 + `npx skills add` 一行安装 = "离开具体的人，方法还能被复用"（WorkBuddy 判据）——一个人的库已经满足组织库的形式条件，差的只是第二个使用者。
- **执行住在 agent 里** = Agent 工作流。loop 自动跑、路由自动选、产物自动归档——人从执行者退居修宪者。

搬家的顺序不可跳：没写成文的原则进不了框架，没框架化的方法编不成 workflow，没有数据回流的 workflow 养不出更聪明的 agent。**你花一周走完了前两步搬家，第三步（真实业务 workflow + 复用记账）是下一程。**
