# Agent 工作区审计

对每个由 AGENTS.md（或 CLAUDE.md）治理的工作区，产出一份证据化评分卡。
核心信念：**协议写得漂亮不值钱，"写下来的系统"与"实际运转的系统"的差距才是审计对象。**

## 铁律

1. **只读。** 审计过程不修改、不移动、不删除任何被审计文件。报告先交用户，改进另行授权。
2. **每个分数必须挂证据。** 没有文件路径、行数、日期、计数支撑的判断不许写进报告。
3. **区分"死规则"与"新规则"。** 规则从未触发，可能因为项目太年轻（正常），也可能写了没人执行（债务）。用项目年龄和活动痕迹判断，不要一律扣分。

## 流程

### 1. 发现目标
`find` 用户指定范围内所有 AGENTS.md / CLAUDE.md。有路由型母文档（管多个子项目的）时，母文档单独算一个审计对象。
同时记录**黑户**：有项目形态（自有目录结构、近期活动、多次复用）但没有协议文档的文件夹。

### 2. 采集证据（每个目标）
- 协议本体：行数、语言（单语/双语）、是否有镜像副本
- 目录契约 vs 实况：协议声明的每个目录是否存在、文件数、最后修改时间
- 执行痕迹：log / manifest / runs 等留痕机制是否在用，条目数与时间跨度
- 卫生：根层散落文件、空目录、临时文件（.DS_Store、未命名*）、命名一致性
- 敏感信息：抽查协议声明的密钥规则是否被遵守（只查是否泄漏，不复述值）

### 3. 五维评分（各 0–10）

| 维度 | 问的问题 | 低分信号 |
|---|---|---|
| A 协议质量 | 目标、边界、目录契约、禁止事项是否清晰可执行 | 只有愿景没有操作规则；边界含糊 |
| B 协议-执行一致性 | 规则是否真的在运转 | 死规则（如巡检从未跑）；被违反的规则（如该登记的没登记） |
| C 工作区卫生 | 文件各归其位吗 | 无主文件、空壳目录、临时文件堆积 |
| D 会话固定成本 | agent 每次上岗要读多少 | 双语双写、镜像双写、只增不减的巨型 log 无摘要 |
| E 可移植与兼容 | 换个 agent/机器还能用吗 | 单一 agent 绑定（无 CLAUDE.md 兼容）、规则真相多处存放 |

锚点：9–10 = 可作模板推广；7–8 = 良好，个别债务；5–6 = 制度或执行有明显洞；≤4 = 对 agent 弊大于利。

### 4. 输出评分卡

每个目标一张，统一格式：

```
## <工作区名>（<角色：路由/项目>）
A×  B×  C×  D×  E×  | 总评 ×.× 
最值得表扬：<一句，带证据>
最大的洞：<一句，带证据>
修复清单：<按성本从低到高，最多 3 条>
```

结尾必须有**横向对比表**和**系统性发现**（多个工作区共有的问题，如全体缺 CLAUDE.md——这类问题修一次惠及全部，优先级最高）。

## 定期复检

本 skill 适合周期运行（如每两周或每次大批量 ingest 后）。复检时先找上次报告，报告差分：哪些洞补上了、哪些新增、分数变化。上次报告默认存放在被审计根目录的 `_audit/` 或用户指定位置。
