# Trigger Audit — 触发设计审计

核心原则：**Skill = Trigger + Action，不是知识文档。** 人不主动搜索技能，是场景线索触发内隐记忆。description 的使命是让 skill 在正确场景"自然浮现"，写功能清单是最常见的死法。

## 审计清单（对每个 skill 的 description 逐项打分，每项 0-2 分）

1. **行为逆向**：是否从"用户会说什么话"出发，而非"skill 能做什么"？
2. **句式覆盖**：是否覆盖四类表达——直接（"画个架构图"）、间接（"给新人讲讲系统结构"）、模糊（"把这个理一理"）、情绪化/困境式（"这话题越聊越乱了"）？含用户**不知道术语时**的说法吗？
3. **负触发**：是否写明什么时候**不该**触发（与相邻 skill 的分界）？
4. **兜底**：Action 是否永不失败（找不到/信息不足时有明确降级路径，而不是硬编）？
5. **可验收**：skill 正文是否含客观断言（第三方可判定输出合格与否）？

## 输出格式

```
## 🔍 触发审计：<skill名>
总分 x/10 ｜ 判级：A(9-10) / B(7-8) / C(5-6) / D(<5)

| 维度 | 得分 | 问题 |
|---|---|---|
（五行）

**重写后的 description**：<直接给可粘贴的成品，不给"建议方向">
**一个测试句**：<一句该触发但当前description可能漏掉的用户原话>
```

## 硬性断言（自检）

- 每个被审 skill 都给出重写成品，不许只批评不动手
- 重写后的 description 含 ≥ 4 条用户原话式触发句
- 测试句必须是"不含 skill 名称和术语"的自然说法

## 边界

- 只审触发与结构，不审领域知识对错。
- 批量审计（整个目录）时先出总分排行榜，再逐个详审得分最低的 3 个，其余仅列分数——渐进式披露，不刷屏。
